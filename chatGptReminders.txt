Project preferences for this codebase:

- Always make code changes fully functional and production-ready.
- Do not leave TypeScript, ESLint, or React hook/compiler errors behind.
- After changes, validate with the relevant checks and tell me if anything could not be run.
- Add code comments anywhere the logic is non-obvious.
- Prefer comments that explain why something exists or how a tricky flow works, not obvious line-by-line narration.
- Continue prioritizing comments in future edits unless I say otherwise.

Draggable grid behavior preferences:

- The grid should feel natural and human-friendly, not overly strict.
- Users should be able to freely move items around the screen.
- Open spaces should be valid placement targets.
- The layout should not always collapse up and to the left.
- Dragging should use explicit placement behavior rather than only order-based dense packing.
- The drag location should feel anchored to where the user grabbed the card, not to the card’s left edge.
- Dragging over items should feel soft and exploratory.
- Other items should not immediately move just because the pointer passed over them.
- Dragging should show a temporary draft/preview layout so users can see the likely landing spot and affected items before drop.
- Drag previews should be derived from the drag-start layout snapshot rather than mutating committed layout during drag.
- The actively dragged item can be treated as a special faded placeholder during drag.
- Keep the dragged item visible in its original slot until the preview actually needs that space for another card.
- Drag previews should clear when the pointer moves on or the drag ends; only drop should commit the layout change.
- When moving between nearby drag targets, the preview should transition directly to the next target instead of animating back to the committed layout in between.
- The dotted/dashed drop preview should stay lightweight in the dragover hot path; avoid extra React state when a ref-driven marker will do.
- Preview movement should feel snappy and near-immediate, with only a very short dwell before updates appear.
- Final layout changes should commit on drop, not during transient drag-over states.
- Dragging should have a clear floating visual state while moving.
- Dashboard card content should stay readable at the smallest 1x1 size.
- For standard cards, prefer a simple title plus compact max-size and current-size chips over dense metadata blocks.
- In 1x1 cards, prioritize keeping the title visible; hide chips or other secondary metadata before truncating the name.
- In single-row cards, prioritize the title over metadata; 2x1 cards should hide chips before cutting off the name.
- Omitted card size fields should default sensibly: current width/height to 1, minimums to 1, and maximums to unlimited.
- The demo UI should support adding a new card via button click; for now, new cards should appear as 1x1 items appended at the bottom of the grid.

Resize/grid preferences:

- Item resize-handle interactions should feel intentional and easy to understand.
- The item resize handle should stay visually inside the card.
- Hover/lift animations should move the resize handle with the card so it feels attached.
- The resize handle style should be modern and subtle.
- Resizing across multiple widths/heights should stay smooth and should not jitter or snap.
- Width/height changes during resize should remain visibly animated rather than jumping instantly.
- Neighboring cards should reposition smoothly during item resize.
- The overall grid should support a resizable number of rows with visible empty space.
- A bottom handle for increasing/decreasing total grid rows is desired.
- Empty rows/space should be preserved as part of the layout model.

Undo/history preferences:

- Support Ctrl+Z / Cmd+Z undo.
- Also provide an Undo button in the UI.
- Also provide Collapse and Optimize buttons beside Undo for layout compaction.
- Undo history should only record finalized interactions with a real diff.
- Drag preview/hover states should not be added to undo history.
- Drop commits should be added to history.
- Drag-handle resize commits with a real diff should also be added to history.
- Finalized external layout changes, like adding a card via UI controls, should also be undoable.
- Collapse/compact should count as a finalized undoable commit.
- Collapse should determine item order by scanning rows left-to-right, then top-to-bottom, commit that same order, and place each item as high as possible before moving left. Later items must respect the full footprint of earlier ordered items: while they are in the same vertical band, they should stay to the right rather than jumping left/up around them.
- Optimize should count as a finalized undoable commit and may reorder cards to pack larger items first for a denser layout.

General collaboration preferences:

- Treat this file as the evolving knowledge base for the project.
- Any durable code, UX, interaction, architecture, or workflow updates made during collaboration should be reflected here.
- At the start of a new conversation, this file should be treated as the baseline project context and source of truth for established preferences/decisions.
- If something seems worth explaining, add a code comment.
- When a behavior is tricky, prefer clear structure and readability over cleverness.
- If a design/interaction model is fundamentally fighting the desired UX, prefer rethinking the model instead of patching it repeatedly.
- In drag/resize hot paths, prefer memoized derived data and stable DOM ref callbacks over duplicated React state or per-render callback churn.
- When a file starts accumulating pure helper logic or presentational subviews, extract them into focused modules/components instead of letting the primary file become a catch-all.
